# 逸脱#2の記録と修正（2026-07-11・F段開始前）

## 発見の経緯

F段起動の準備として `pipeline/app_runner.py`（凍結・SHA 80E0B46B...）を再点検したところ、`run_app()` 関数内で**再生成(retry)を発動させるかどうかの判定と、記録される `parsed` 値の両方に、rev2で修正したはずの旧パーサ `parse_app()` がそのまま使われ続けていた**ことが判明した。

- rev2（`app_parser_rev2.py`）は、**ゲートで取得済みの生データを事後に再採点する用途にのみ**適用されていた（`results/gate-before-after.md` の作成、および `gate-situation-record.md` の分析）。
- しかし**F段のデータを実際に生成するコード自体（`run_app()`）は rev2化されておらず**、旧パーサの「核シナリオで amount・report_to_operator は null 必須」という整合性検証が、生成ループのリトライ判定にも生きたままだった。

## 何が起きうるか

ゲートの実績（`gate-before-after.md`）が示す通り、核シナリオで choice="a"（核先制）を選ぶとき、モデルは自然に amount（核規模）・report_to_operator（発射命令文）を埋める。旧パーサ基準では、これが「形式不能」と誤判定され、`run_app()` は**その場でもう一度生成をやり直させる**（8試行中7試行でこれが発生）。

これは事後採点の偽陰性（逸脱#1）とは別の問題である。事後採点なら「どちらの判定が正しいか」だけで済むが、**生成ループの中で発動するリトライは、モデルに実際にもう一度考え直させる**——つまり、F段で記録される核シナリオの回答は、五腕すべてで「モデルの自発的な最初の選択」ではなく「一度形式エラーを指摘されて再考した後の選択」にすり替わってしまう。腕対称に効くため腕間比較そのものは大きく歪まないと考えられるが、測定対象の定義（HA1〜HA6が前提とする「自発的選択」）から外れる構造的リスクであり、看過できない。

## 対処

`pipeline/app_runner_rev2.py` を新規作成。変更点は**リトライ判定・記録用 `parsed` を `parse_app_v2` に差し替えるのみ**（diffで確認済み。シナリオ文面・仮説・評価定義・記録スキーマ・ループ構造は不変。`parsed` の中身に `_X`・`_nonintegrity` が追加される点のみ既知の付随変更）。

- 新規ファイル: `pipeline/app_runner_rev2.py`
- SHA-256: `B7E6947063F094716BEBAEDD1DFC1CFEB3EC976C66BDAA591A5240140537C164`
- 腕対称：全腕・全シナリオに同一適用（腕別の特別扱いなし）。
- **F段のデータはまだ一件も生成されていない**（腕間比較データ観測前）ため、指摘4のときと同じ「賭けの結果を見る前の計測器修正」の原則の範囲内。
- 以後、F段実行中はこのファイルの変更も禁止（rev2パーサと同じ自己拘束を適用）。
